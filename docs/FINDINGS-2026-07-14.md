## 0. The structural theme: a whole tier is built but not wired into the running server

This reframes ~10 findings, so it comes first. `src/server/index.ts` boots via `createProductionComposition(environment, {docsRoot, webRoot})` — with **no** `resources.github` / `resources.outline`. `dependencies.ts` never imports `github-production-composition.ts` or `production-content.ts`, and all connector route mounting plus the workflow scheduler loop are gated on those resources. So in the deployed server:

- **GitHub and Outline routes are unmounted** (`state: DISABLED`). The connectors' code exists and is unit/fixture-tested, but the HTTP surface 404s. This is likely *intentional staging* (dedicated composition files exist; recent commits add "live GitHub App preflight" and "live Outline provider verification"). The consequence for prioritization: the connector bugs below (F-11, F-12, F-15, F-20, F-24) are **real defects but currently latent** — fix them *before* flipping the wiring on, not after.
- **The workflow scheduler interval loop never starts.** `scheduler.start()` (the `setInterval(() => engine.tick())` driver) is called from exactly one place — `github-production-composition.ts:116` — which is itself never invoked by default. `dependencies.ts:1269` runs a single `workflowScheduler.tick()` at boot and nothing more. **(C-4)**
- Coupling the scheduler loop to the *GitHub* composition is itself an architectural smell: wiring Outline-only, or Foundation-only automation, still leaves the loop dead.

Several fully-built modules likewise have **zero production callers** and read as implemented capability during review: run/orphan reconciliation (F-27), the operation-authorization consumer + fencing enforcement (F-30), the delivery ack-timeout machinery (F-31), the SSE projection handler (F-9, never mounted — the browser `EventSource` 404s forever), and three orphaned GitHub helpers (F-45). Per the repo's own single-implementation rule, each should be either wired or deleted.

## 1. Critical

**C-1 — Registration-policy schema-verify bricks the server the moment an owner changes registration mode.**
`src/server/db/migrations/0018_email_registration_policy.verify.ts:40` hard-fails unless `mode === "INVITE_ONLY"`. But `validateClaimedSchema` runs this on **every boot** (`migrate.ts:256-259,183`), and `registration-policy.ts` `updateMode()` lets an owner set `CLOSED`/`ALLOWLIST` at will. `migrate()` is called with no try/catch (`dependencies.ts:267`, `index.ts:23`, `production-bootstrap.ts:82`). So: owner switches mode → next restart (deploy/crash/respawn) → uncaught `SCHEMA_INTEGRITY_INVALID` during dependency construction → server never boots, with no code path back to `INVITE_ONLY`. Permanent brick until someone hand-edits SQLite. **Runtime-reproduced:** fresh `migrate()` (seeds `INVITE_ONLY`) → `UPDATE auth_registration_policy SET mode='ALLOWLIST'` → second `migrate()` throws `SCHEMA_INTEGRITY_INVALID`. **Fix:** verify schema *shape* only (or that exactly one row with a valid mode exists), not the seed value; assert seed state once at migration-apply time, not on every boot. Add the repro above as a regression test.

**C-2 — Any transient token-issuance failure permanently stops runner reconnection.**
`src/runner/transport/wss-client.ts:411` catches *any* throw from `issueAccess()` — a 5xx during a deploy, a DNS blip, a fetch timeout, a JSON parse error — and maps it to the terminal `AUTHENTICATION` disconnect reason, which `reconnect.ts` sets to `STOPPED` with no further attempts. `production.ts` calls `client.start()` once with no watchdog. The daemon keeps running but goes silently offline forever until an operator restarts it — defeating the entire backoff state machine for its most common failure mode. **Fix:** distinguish genuine 401/403 from transient (network/5xx/timeout) failures; only the former is terminal, the rest retry.

**C-3 — Runners can self-report gate outcomes for unapproved manifests on the live evidence path.**
`execution-authority.ts:2298` (`executeEvidence`, the wired `RECORD_EVIDENCE` handler) copies runner-supplied `gateKey`/`manifestFingerprint`/`outcome` straight into `run_evidence` for `GATE_EVALUATION`, with only run-revision + assigned-runner checks — **no** `isGateFingerprintApproved` call. The module built to enforce that trust boundary (`gates/fingerprints.ts`, `gates/evaluations.ts`) has zero importers outside `dependencies.ts`, and its coordinator is never read by any route. So any authenticated (or compromised) runner can record `outcome: PASSED` for a gate that was never approved or executed, and anything downstream trusting those rows trusts an unverified claim. **Fix:** route the `GATE_EVALUATION` case through `gates/evaluations.ts` validation before persisting, and delete the now-parallel unused implementation.

**C-4 — The workflow scheduler loop never runs, so executions stall after the first boot tick.**
See §0. The workflow routes *are* mounted by default (unlike the connectors): `app.ts:103` gates them on `options.automation`, which `dependencies.ts:994` always populates via `?? { workflows, templates, runtime }` — so `POST /api/v1/workflow-executions` is reachable. But `engine.tick()` is the sole path that sets `dispatched_at` on `workflow_launch_intents` (launches the next step), fails past-deadline executions, and drains `workflow_cancellation_outbox`, and `dependencies.ts:1269` runs it exactly once at boot (`scheduler.start()` only fires inside the unused GitHub composition). So any execution created after boot never dispatches subsequent steps, never enforces its deadline, and never processes cancellations — **live, not latent.** This is more severe than — and underlies — the workflow-engine logic bugs in F-1, which implicitly assume `tick()` recurs. **Fix:** start the scheduler loop from the default composition (decoupled from the GitHub wiring), and add an integration test that creates an execution and asserts a later step dispatches.

## 2. High severity

### Backend & workflow logic

**F-1 — One `accept()` ternary in `workflow-engine.ts` carries three defects.** (`workflow-engine.ts:1236-1259`)
- **(a) ANY-join remainder clobber → permanent hang.** After an ANY join commits, `evaluateJoin` returns no `transition`, so `targetKey` stays undefined and the else-branch forces `state=WAITING` at the already-consumed branch node. Under `LET_FINISH` (guaranteed) or a `CANCEL_REMAINDER` race, a completing remainder branch resets the execution — and if the join's target was a `HUMAN_DECISION`, `decide()` then rejects the operator forever (`current_node_key` no longer matches). Workflow sits until its deadline fails it.
- **(b) Branch completion while PAUSED silently un-pauses.** The PAUSED-preserving branch only runs when `targetKey` is truthy; a not-yet-satisfied join yields no target, falls to the else, and forces `WAITING` regardless of `current.state`. Pausing a workflow with in-flight parallel branches doesn't actually hold it.
- **(c) `maximumRunCount` hit mid-transition strands the occurrence.** `accept()`'s bare `catch` maps `WORKFLOW_RUN_BOUND_EXCEEDED` to `WORKFLOW_STORAGE_FAILED` and rolls back the occurrence's TERMINAL write + event receipt, leaving it `RUNNING` with its result lost — unlike the cycle-bound path which fails cleanly. **Fix:** leave a committed join untouched; preserve `PAUSED` when there's no transition; handle the run-bound like the cycle bound (commit TERMINAL, fail the execution).

**F-2 — Coordination-record coalescing bypassable via a stale alias id.** (`coordination-records/registry.ts:46`, `execution-authority/persistence.ts:411`)
`resolveCoordinationForLaunch` selects by the client-supplied `coordinationRecordId` verbatim; `canonicalCoordinationRecord()` (which resolves alias→canonical) has **zero callers**. After two records coalesce, a client still holding the old alias id can launch a new mutating run: the guard check queries HELD guards by the alias id, but coalescing already moved that guard to the canonical id — so the launch sees no guard and proceeds. Two concurrent mutating runs on one work item, with no branch-collision detection. **Fix:** resolve the selection through `canonicalCoordinationRecord()` before the guard query; reject/redirect known alias ids.

**F-3 — Headless output drain drops data and retry-storms when the socket is down.** (`runner/headless-output.ts:91`)
`drain()`'s loop has no `catch`. `sendText` throws whenever `reconnect.state !== ACTIVE` (routine during reconnect), but the `finally` already decremented bookkeeping — so the chunk's text is lost, never requeued. `push()` invokes via `void drain()`, discarding the rejection, and the `.finally` re-invokes `drain()` immediately, producing a tight unhandled-rejection loop for every queued item — which in Bun can terminate the daemon. **Fix:** catch inside the loop, requeue-or-drop deliberately with backoff, and surface send failures instead of `void`.

### Auth & HTTP security

**F-4 — Preauth rate limiting shares one global bucket, enabling a trivial login lockout.** (`dependencies.ts:112`)
Every unauthenticated flow passes the literal `actorId="PREAUTHENTICATED"`, so all clients share **one** 120-req/60s bucket per path. One client hammering `POST /api/auth/email-otp/verify` exhausts it for everyone, and the OTP handler then returns `EMAIL_OTP_INVALID` (not 429) — so a legitimate user with the correct code is told it's wrong. **Fix:** key the preauth bucket per source IP/connection, and return 429 on trip.

**F-5 — Email OTP rotation doesn't deterministically invalidate the prior code.** (`identity/better-auth.ts:617`; enroll/change-email path `:987`)
`resendStrategy: "rotate"` is configured, but better-auth 1.6.23's rotate path inserts a new `auth_verifications` row **without deleting** the old one, and both reads select `ORDER BY createdAt DESC LIMIT 1` with no tiebreak. `createdAt` is a millisecond integer, the table has no unique index on `identifier`, and when two resends land in the same millisecond the tie resolves to the *older* row — so the superseded code still verifies. The structural weakness was reproduced independently by inserting two rows with an identical `createdAt` and observing the `ORDER BY createdAt DESC LIMIT 1` read return the *older* one. The test `better-auth-email-otp.test.ts:480` is **flaky under full-suite load only** (green 12/12 in isolation, red once in the 214-file run), consistent with the collision manifesting under load rather than a deterministic race — don't treat the isolated green run as proof it's fixed. The same hole applies to the change-email/enrollment OTP path. **Fix:** inside `withEmailSendLock`, delete existing rows for the identifier before `sendVerificationOTP` (do it once, structurally), and add a UNIQUE index on `auth_verifications(identifier)` — this makes invalidation deterministic regardless of timestamp precision.

**F-6 — GitHub mutation surface skips the project-membership authorization every read path enforces.** (`connectors/connector-authority.ts:989`) *(latent — route unmounted, fix before wiring)*
`POST /api/v1/github/mutations` → `mutateAsMember` authorizes only an *active session for any member* plus a READY connector scope; it never checks the actor belongs to `command.projectId`. Every read surface gates on `authorizeProject(actor, projectId)` (`github-planning.ts:41`, `outline-*`). Where more than one project is mounted, any member could mutate another project's issues/PRs. **Fix:** add `authorizeProject(actor, command.projectId)` to the mutation path; ship the production `authorizeProject` (currently test-stubbed) alongside.

### Frontend (emphasized)

**F-7 — No run history or evidence viewer, though the endpoints are fully built.** (`web/features/runs/runs-feature.tsx:70`)
The Runs page only ever `POST`s to `/api/v1/runs`; it never calls `GET /:runId` or `/:runId/evidence` (both implemented), and `cancel`/`resume` are unreachable. "Recent runs" shows only the run created in the current session and reverts to "No runs yet" on reload — the core object of the product has no read surface. **Fix:** list runs on mount; add a run-detail view fetching evidence; wire cancel/resume.

**F-8 — Run creation swallows failures and renders error codes as fake QUEUED/COMPLETED runs.** (`runs-feature.tsx:108`)
`create()` has no try/catch and is called via `void`, so `browserJson`'s throw on any non-2xx is silently discarded (modal just sits). And a domain `{ok:false}` sets `result = error.code`, which then renders as a run row with a hardcoded `QUEUED` status **and** increments the COMPLETED counter. A rejected run is shown to the operator as queued-and-completed. **Fix:** try/catch + explicit error state; distinguish success / domain-error / HTTP-error; don't derive counts from an error code.

**F-9 — New-run form posts hardcoded `project_1`/`repository_1`/`preset_1`.** (`runs-feature.tsx:22`)
Real IDs are `${prefix}_${randomBytes(24).base64url}` (`dependencies.ts:143`); the literals never exist. There's no repository/preset picker, so the primary action can only succeed against a fixture-seeded DB. **Fix:** fetch the caller's real project/repos/presets into selects; remove the literals.

**F-10 — GitHub planning is hardwired to `project_1` and always shows "unavailable".** (`github/github-planning-feature.tsx:18`)
Baked-in project id + no `response.ok` check before a `.strict()` parse means any non-success body throws into the catch → permanent "unavailable" (and the route 404s in the default deploy anyway, per §0). The sidebar "PROJECT: Collab" is static text with no switcher. **Fix:** drive the project id from a real fetched project / route param + a switcher; check `response.ok` before parsing.

**F-11 — The new-run modal has `role=dialog`/`aria-modal` but no focus management at all.** (`runs-feature.tsx:86`)
No focus-on-open, no Escape, no focus trap, no focus restore, no backdrop dismiss (grep: no `useRef`/`focus`/`onKeyDown`/`Escape`). `aria-modal="true"` actively lies to assistive tech on the app's primary creation flow. **Fix:** use a native `<dialog>` with `showModal()` (trap/Escape/backdrop for free) or add the handlers manually.

**F-12 — The three screens wired to *real* data render unstyled.** (`github-planning-feature.tsx:33`, `inbox/index.tsx:17`, `command-center/index.tsx:12`)
GitHub/Inbox/Command Center use `feature-page`/`page-header`/`planning-board` — classes that don't exist in `styles.css` (the only stylesheet) and aren't Tailwind utilities. The mock screens (Runs/Members/etc.) use the defined `workspace-header`/`summary-grid`/`data-panel`. So the genuinely-functional pages look broken while the placeholders look polished. **Fix:** reuse the existing styled classes (smaller diff) or define the missing ones.

**F-13 — "Invite member" posts to a route that doesn't exist.** (`web/features/members/members-feature.tsx:15`)
`POST /api/v1/members/invitations` has no server route (only `/invitations/exchange|auth/*` exist), and the `invite()` identity-authority method has zero HTTP callers. The form sits beside the correctly-wired email-enrollment and registration-policy panels, so the page reads as functional while this control 404s on every use. **Fix:** wire an invitations-create route to `invite()`/`CreateInvitationSchema`, or disable the form until it exists.

**F-14 — Workflow runtime has no browser entry point.** (`web/features/workflows/execution.tsx:134`)
`POST /api/v1/workflow-executions` + pause/resume/cancel/decisions/events are all mounted, but grep for `workflow-executions` in `src/web` returns zero. `/workflows/journey` is a hardcoded local-state phase machine rendering `"Live canonical proof: BLOCKED"` as a static string. Self-labeled fixture (acknowledged), but the entire runtime-control surface is absent. **Fix:** replace the fixture with a view that drives the real endpoints.

## 3. Database

**F-15 — OTP secret rows are never garbage-collected in email-OTP-only deployments.** (`identity/better-auth.ts:750`)
The only opportunistic `DELETE ... auth_verifications WHERE expiresAt <= ?` is gated behind two *passkey* endpoints. A passkey-free deployment never triggers it, so every OTP send/rotate leaves a permanent row holding a hashed OTP secret tied to an email; and there's no index on `expiresAt`, so the rare cleanup is a full scan. **Fix:** run the cleanup on the email-OTP paths (or any auth request) and index `auth_verifications(expiresAt)`.

**F-16 — OTP rate-limit cap is duplicated between a schema CHECK and an app constant.** (`0018_...sql:61` vs `better-auth.ts:19`)
`CHECK(send_count BETWEEN 1 AND 3)` hardcodes the same `3` as `EMAIL_OTP_SENDS_PER_MINUTE`. Raise the constant without a migration and the increment `UPDATE` throws a CHECK violation *inside* the transaction, uncaught out of `requestEmailOtp` — turning a routine tweak into an unhandled 500 on every 4th request. **Fix:** widen the CHECK generously or wrap the call site to treat any failure as "rate limited".

## 4. Backend logic (latent connector paths + telemetry)

**F-17 — Outline patch application corrupts the wrong location and rejects valid patches.** (`outline/production-content.ts:153`) *(latent — route unmounted)*
`applyPatch` concatenates only `-`/`+` lines and does `body.replace(removed, added)` — ignoring the `@@` hunk header and context lines the contract requires. `String.replace` with a string arg hits only the **first** occurrence, unanchored. Reproduced: a hunk targeting the second repeated marker edits the first; valid pure-insertion and multi-context hunks return `OUTLINE_PATCH_INVALID`. All *after* the exact-revision CAS passes — false safety, silent wrong-region edit returning success. **Fix:** apply by hunk offset + context matching; honor insertions; add repeated-line / insertion / multi-hunk tests.

**F-18 — Outline edit CAS is read-check-then-unconditional-write (TOCTOU).** (`production-content.ts:321`) *(latent)*
The precondition is checked against an earlier read, then `documents.update({id,text})` writes with no revision/conditional param (Outline is last-write-wins). A human edit landing in the window is silently overwritten, call still returns success — while the GitHub side tags equivalent windows `RESIDUAL_RACE`. **Fix:** re-read and verify the revision advanced exactly, or pass Outline's optimistic-concurrency param; at minimum emit a `RESIDUAL_RACE` marker.

**F-19 — Reconciliation scan uses throwing `.parse()` with no per-record isolation.** (`github/rest-provider.ts:543`) *(latent)*
One record GitHub returns that the schema doesn't anticipate (new `state_reason`, over-long field, unexpected null) throws out of the async generator and aborts the whole connector's pass; the cursor never advances, so it can get stuck re-hitting the poison record. **Fix:** try/catch per record; skip or emit REDACTED/UNAVAILABLE and continue.

**F-20 — Workflow usage telemetry is unwired and conflates zero-attempts with COMPLETE.** (`telemetry/workflow-usage.ts:20`)
`record()` is never called on a reachable path, and `aggregateWorkflowUsage` computes `status = known.length === attempts.length ? COMPLETE : PARTIAL` with no NONE case — so an empty `attempts` array reports `COMPLETE` with confirmed-zero usage, the exact zero/UNKNOWN conflation the sibling `telemetry/usage.ts` avoids. **Fix:** wire `record()` into the attempt/gate completion path and return NONE when `attempts.length === 0`.

**F-21 — Headless live output is buffered on ingest but never cleared per-attempt and never read out.** (`wss/durable-dispatch.ts:184`, `live-output.ts:184`)
`LiveOutputHub.clear()` has zero call sites (only full-shutdown `clearAll`), so per-attempt target structs accumulate; and `createRunnerChannel` is built without `liveOutput`, and the returned `output` object has no consumer — so headless output is ingest-only, buffered then discarded with no egress. **Fix:** `clear('ATTEMPT', id)` on terminal state; wire an `output.inspect` consumer (or confirm egress is tracked-pending).

## 5. Runner (long-lived-process hazards)

**F-22 — No crash-recovery reconciliation is wired into the production runner.** (`runner/production.ts:472`)
`createRunnerDaemon` (which calls `reconcile()` on startup, and is tested) is never imported by `production.ts` — the actual LaunchAgent entry builds its own `start()` with no reconciliation. If the daemon crashes after `recordStarted` persists a `STARTED` row but before the `PROCESS_STARTED` event is queued, `reserve()` returns `RECONCILE` forever (nothing transitions a `STARTED` row out on restart) and any surviving child process is untracked — never cancelled, never reaped, its worktree still claimed. **Fix:** run startup through `createRunnerDaemon` with a real `reconcile()` that inspects/kills stale `STARTING`/`STARTED` rows and resets them.

**F-23 — Four unbounded in-memory maps in long-lived processes.**
Each grows one entry per run/attempt for the process lifetime with no eviction: `production-composition.ts:220` (`launches`/`revisions`/`worktreePaths`), `wss/durable-dispatch.ts:68` (`permits`/`activeOutputTargets` — permit strings are the larger cost), `enforcement/trusted-host.ts:11` (`sessions`), and the live-output targets in F-21. On a 7-day dogfood these leak steadily. **Fix:** delete entries when the attempt/outbox reaches a terminal state.

**F-24 — Runner credential secret is passed as a CLI argv to `security`.** (`runner-store.ts:41`)
The full serialized `StoredRunnerCredential` (private key, runner credential, pairing secrets) is an argv element to `/usr/bin/security add-generic-password -w <secret>`, visible via `ps`/process listing for the spawn's lifetime. **Fix:** write the secret to the process's stdin instead of argv.

## 6. Frontend improvements & hardening

- **F-25 — No in-flight guard on run/invite submit** (`runs-feature.tsx:135`, `members-feature.tsx:55`): double-click fires two POSTs with two fresh `crypto.randomUUID()` idempotency keys, defeating server dedup → two real runs / two invites. The setup/recovery forms already gate on `disabled={state==="WORKING"}`. **Fix:** mirror that pattern.
- **F-26 — Members roster is hardcoded** (`members-feature.tsx:33`): "Tim Kraus / 1 owner / 1 passkey" is literal JSX with no roster fetch; invited members never appear even though the invite path *would* work. Actively misleading. **Fix:** add and fetch a roster endpoint; drop the literals.
- **F-27 — Runs counts fabricate state** (`runs-feature.tsx:53`): ACTIVE/WAITING hardcoded to 0; a fresh run is counted COMPLETED ("Committed history") while shown as QUEUED. Plus the SSE stream delivers full `RUN_CHANGED` run payloads that the UI discards, updating only a cursor string. **Fix:** drive counts + list from a runs endpoint and upsert `RUN_CHANGED` payloads.
- **F-28 — Projection SSE client has no `onerror`** (`web/projection-client.ts:9`): on a permanent close (e.g. 401 after session expiry) `EventSource` stops silently; the "live" badge freezes on a stale cursor with no signal. **Fix:** surface connection state; show reconnecting/disconnected.
- **F-29 — Inbox & Command Center use raw `fetch` with no zod and map errors to `[]`** (`inbox/index.tsx:11`, `command-center/index.tsx:6`): a backend outage or contract drift renders as "nothing needs attention," and untyped payloads are trusted directly. **Fix:** route through `browserJson` with a schema; distinct error state; add loading + unmount guards.
- **F-30 — Outline search: no error/loading/in-flight state** (`outline/search/index.tsx:18`): failures look like empty results; rapid searches race with no ordering guarantee. **Fix:** loading + error states, disable while in flight, AbortController/latest-query guard.
- **F-31 — Dead components shipped** (`templates/workflow-bindings.tsx`, `outline/editor`, `outline/connection`, `components/ui/button.tsx`): built, typed, never imported. The Outline pieces are a real gap (no UI to connect/revoke a workspace or create/edit a doc with the conflict UX); the Button + WorkflowBindings are pure dead weight. **Fix:** wire the Outline pieces; delete the rest.
- **F-32 — Workflow Studio Validate/Publish are inert** (`workflow-studio/editor.tsx:29`): no `onClick`, validation panel hardcoded to `[]`, drafts never `POST` to `/workflow-drafts/:id`. Looks like a working authoring tool; loses everything on reload. **Fix:** wire Publish + real diagnostics, or disable until wired.
- **F-33 — Runner fleet + GitHub write have no UI** (`runners-feature.tsx:70`, `github-planning-feature.tsx:18`): "Pair runner" has no handler and the fleet list is unconditionally empty even after a successful pairing; the GitHub mutation endpoint has no controls. **Fix:** fetch real fleet state; add pairing/profile/mutation controls.
- **F-34 — Every sidebar nav is a full-page reload** (`web/app.tsx:89`): plain `<a href>` with no `pushState`/`popstate`, so each click reboots React and re-runs `useSession()`, flashing "Checking session…" and losing in-memory state. **Fix:** ~15 lines of `pushState` + `popstate` + `useState(pathname)`.
- **F-35 — Setup-progress labels fail WCAG AA contrast** (`styles.css:458`): `#8a8c93` on white ≈ 3.35:1 (< 4.5:1); `--muted` on `--bg` is borderline ~4.48:1. **Fix:** darken to ~`#6b6d75`.

## 7. Other backend/security hardening

- **F-36 — 64 KB body limit bypassable via omitted Content-Length** (`http/middleware/request-limits.ts:28`): the pre-check trusts the header (absent → 0 → passes), then `req.text()` buffers the whole chunked body before the real check; no `maxRequestBodySize` is set, so the ceiling is Bun's ~128 MB default. Cheap memory-pressure DoS. **Fix:** enforce the limit while streaming, and set an explicit low `maxRequestBodySize`.
- **F-37 — Unauthenticated secret-exchange endpoints have no rate limiting** (`runner-pairing.ts:103`, device-auth, better-auth bootstrap/invitation): high-entropy secrets make brute force impractical, but the missing throttle is an inconsistent defense-in-depth gap. **Fix:** apply the (per-IP) preauth limiter uniformly.
- **F-38 — Member revocation doesn't purge change-email OTP rows** (`identity/revocation.ts:132`): it reconstructs `change-email-otp-<email>` but the stored identifier is `change-email-otp-<old>-<new>`, so the row survives to its 5-min expiry. Low (gated behind a still-live browser session), but breaks the clean-up-on-revoke invariant. **Fix:** match with a `LIKE ...-%` pattern.
- **F-39 — Production file-path guard covers only the Resend key** (`shared/environment.ts:127`): `DEPLOYMENT_MASTER_KEY_FILE` (gates the master encryption key), `BOOTSTRAP_SECRET_FILE`, `OUTLINE_TOKEN_FILE` get no `isAbsolute` check. Low today (same cwd), but an inconsistent trust posture. **Fix:** apply the check uniformly.
- **F-40 — Stop-policy: multiple `CONSECUTIVE_MATCHES` share one counter** (`workflows/stop-policy.ts:55`): sibling matchers under one ALL/ANY read/overwrite the same `matches` scalar; a nested one resets to 0 every eval. Latent (no UI builds such a policy). **Fix:** key each node's counter, or forbid multiplicity/nesting in validation.
- **F-41 — `durable-dispatch` rebuilds `ConfigurationPersistence` per outbox row** (`durable-dispatch.ts:101`): pure allocation waste on the priming/resend hot path. **Fix:** hoist the facade out of the loop.
- **F-42 — Orphaned GitHub helpers with latent bugs** (`github/reconciliation.ts:40`): `reconcileGitHubScope`, `GitHubInstallationTokenCache`, and `performRevisionCas` are superseded by the injected `reconcile` and inline CAS — dead in production (only their own tests exercise them). The reconcile summary also miscounts `updated` and never advances its cursor. **Fix:** delete, or wire and fix the counting/cursor.

## 8. Unwired capability modules (confirm tracked or delete)

Each is fully built with zero production callers — see §0. Listed so the wiring gap can't hide as "done":

- **F-43 — Run/orphan reconciliation** (`modules/runs/reconciliation.ts`): lost-runner detection + orphan quarantine, never invoked; also has a latent `elapsed === offlineSeconds` exact-equality `MARK_OFFLINE` bug (use `>=`).
- **F-44 — SSE projection handler never mounted** (`http/sse.ts:173`): `createProjectionSseHandler` is imported nowhere; `/api/v1/events` 404s and the browser `EventSource` retries forever. **And** even when mounted it has a real bug (F below).
- **F-45 — SSE idle-reset orphans the waiter** (`http/sse.ts:89`): `reset()` splices a RESET into the queue then calls `finish()`, whose drain loop is gated on `queue.length === 0` — so a caught-up idle client parked in `next()` never receives `AUTHORITY_CHANGED`/`STREAM_INVALID`/`SLOW_CONSUMER`; the connection stalls open (no auto-reconnect), leaving a stale, possibly over-privileged view. **Fix:** deliver the RESET through `push()` so it reaches a pending waiter.
- **F-46 — Operation-authorization consumer/fencing is dead outside tests** (`execution-authority/fencing.ts:159`): the issue side mints single-use `operation_authorizations`, but the consumer that marks them CONSUMED while re-verifying fences has no production caller — the anti-replay/TOCTOU checkpoint is inert.
- **F-47 — Delivery ack-timeout machinery is inert** (`wss/production.ts:114`): `onAcknowledgementTimeout` is never wired and `sweepAcknowledgementTimeouts` never scheduled, so a runner that receives but never acks a `LAUNCH_ATTEMPT` isn't proactively detected within the connection's life.

## Coverage & blind spots

Honest limits of this pass, so you know what *isn't* covered:

- **No runtime execution of the frontend.** Findings are from reading every `src/web` file, not from driving the UI. The E2E suite (`test:e2e`) and a real browser session weren't run — recommended as the next step, especially to confirm F-7…F-14.
- **Connector live paths unexercised.** Because the GitHub/Outline routes are unmounted by default (§0), the connector findings (F-6, F-17…F-19, F-42) were verified by static reads + small repros, not against a running connector. They become live risks the moment the composition is wired.
- **The one failing test is flaky** (F-5), so CI will be intermittently red on it until the OTP invalidation is made deterministic; don't treat a green run as proof it's fixed.
- **Thin areas.** The CLI (`src/cli`), the MCP adapter surface, and the backup/restore/key-rotation operations got only incidental coverage; a focused pass there is likely to surface more. Concurrency/fault behavior (restart-during-transaction, outbox replay ordering) was reasoned about but not fault-injected.
- **Severity is current-state.** Latent connector/SSE findings are rated by their real defect; their *live* risk is gated on the wiring in §0.
